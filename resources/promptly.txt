Now I have the complete picture. Here’s the answer:

“NVD refs” is not a vague label — the actual URLs are already being captured. _extract_reference_tags() returns a list called exploit_urls containing the real reference URLs (e.g. https://www.exploit-db.com/exploits/12345). These are stored on the CVE dict as exploit_urls.

The problem is what happens next. When building exploit_sources for the output row, the code does:

if has_nvd_exploit:
    exploit_sources.append("NVD refs")   # ← discards the actual URLs


It throws away cve.get("exploit_urls") entirely and replaces the real links with the generic string "NVD refs". The Vulners path does better — it passes through actual source names like exploit-db or packetstorm.

The fix is straightforward — use the actual URLs instead:

Before:

exploit_sources: List[str] = []
if has_nvd_exploit:
    exploit_sources.append("NVD refs")
if vulners_exploits:
    sources = set(e.get("source", "") for e in vulners_exploits)
    exploit_sources.extend(s for s in sources if s)


After:

exploit_sources: List[str] = []
if has_nvd_exploit:
    nvd_urls = cve.get("exploit_urls", [])
    if nvd_urls:
        exploit_sources.extend(nvd_urls)
    else:
        exploit_sources.append("NVD refs")
if vulners_exploits:
    sources = set(e.get("source", "") for e in vulners_exploits)
    exploit_sources.extend(s for s in sources if s)


This means the Exploit Sources column and the log output will show the real URLs — e.g. https://www.exploit-db.com/exploits/44813; https://packetstormsecurity.com/files/152014 — instead of the generic NVD refs label. The fallback to "NVD refs" is kept for the rare case where NVD flagged a reference as an exploit but the URL somehow wasn’t captured.
